# Web 页面健壮性测试平台方案设计（评审稿）

## 1. 目标与背景

你要做的是一个 **Web 端健壮性测试平台**，核心能力是：

1. 同时打开一组 `iframe`，并发加载并测试一组页面。
2. 通过录制用户操作生成可回放脚本。
3. 按计划反复执行（长时间运行），持续监控稳定性和运行状态。

这份文档只定义设计方案与落地路径，不涉及代码实现。

## 2. 需求拆解

### 2.1 功能需求

1. 支持测试任务（Task）管理：创建、启动、暂停、停止、重试。
2. 支持测试场景（Scenario）管理：包含页面列表、操作序列、断言规则。
3. 支持录制操作：点击、输入、滚动、等待、切换页面、简单断言。
4. 支持回放策略：固定次数、无限循环、按时间窗口执行。
5. 支持并发执行：多个页面在多 `iframe` 同时运行，并支持任务启动时手动指定并发数。
6. 支持实时监控：成功率、失败原因、运行时长、资源占用、心跳。
7. 支持结果归档：每次执行日志、错误栈、事件轨迹（截图/录像在二期补充）。

### 2.2 非功能需求

1. 稳定性优先：长时间运行不内存泄漏、不明显漂移。
2. 可观测性优先：任何失败都能定位到步骤级别。
3. 可扩展性：后续可接入更多页面、断言类型、告警渠道。
4. 安全与隔离：避免测试脚本污染控制台页面。

## 3. 范围边界（先做什么）

### 3.1 V1 建议边界

1. 优先支持同源页面（或可注入测试 SDK 的页面）。
2. 优先支持基础事件录制与回放（点击/输入/等待/断言文本）。
3. 优先支持单机部署、单用户或小团队使用。
4. 先做页面级健康检查与功能主路径回归，不做复杂业务编排。

### 3.2 V2 扩展方向

1. 跨域页面支持（通过 `postMessage + 页面内 Agent SDK`）。
2. 分布式执行器（多浏览器实例/多机器并发）。
3. 告警渠道接入（企业微信/飞书/Slack/邮件）。
4. 更复杂断言（DOM 结构、网络请求、性能阈值、视觉比对）。
5. 失败取证增强（截图/录像）。

## 4. 总体架构

## 4.1 架构分层

1. 控制台层（Web UI）
- 管理任务、场景、执行计划、实时看板。

2. 编排层（Orchestrator）
- 负责任务调度、并发分配、重试策略、生命周期管理。

3. 执行层（Runner）
- 管理 `iframe` 池、加载页面、执行步骤、回收资源。

4. 录制层（Recorder）
- 捕获用户操作并序列化为脚本（JSON DSL）。

5. 监控层（Monitor）
- 采集心跳、步骤耗时、错误码、失败截图与日志。

6. 存储层（Storage）
- 保存任务定义、场景脚本、执行历史、统计指标。

## 4.2 简化数据流

```text
用户在控制台配置任务
  -> Orchestrator 生成执行计划
  -> Runner 分配 iframe 并并发执行
  -> Replayer 按步骤驱动页面
  -> Monitor 实时上报状态
  -> Storage 记录结果
  -> 控制台展示看板与历史
```

## 5. 关键模块设计

## 5.1 iframe 池管理器（Iframe Pool Manager）

1. 维护固定大小的 `iframe` 池（如 4/8/16 个并发槽位）。
2. 每个槽位维护状态：`idle | loading | running | failed | recycling`。
3. 每轮执行后进行槽位回收：清理定时器、解绑监听、重建 `iframe`。
4. 支持“软重置”与“硬重建”两级恢复策略。
5. 并发数支持手动选择（如 1~16），系统按设备能力给出推荐值和上限保护。

## 5.2 录制器（Recorder）

1. 录制事件类型：`click`、`input`、`select`、`scroll`、`wait`、`assert`。
2. 每个事件记录：目标定位、参数、时间戳、上下文页面。
3. 选择器策略：`data-testid` 优先，`id` 次之，最后 `css/xpath`。
4. 自动去噪：合并高频输入事件，过滤无意义鼠标移动。

示例事件：

```json
{
  "stepId": "step_12",
  "type": "click",
  "target": {
    "primary": "[data-testid='login-btn']",
    "fallback": ["#login", "button[type='submit']"]
  },
  "timeoutMs": 5000,
  "retry": 1,
  "meta": {
    "url": "/login"
  }
}
```

## 5.3 回放器（Replayer）

1. 按步骤执行，支持前置等待（元素可见/可点击/网络空闲）。
2. 每步支持超时与有限重试。
3. 失败时产出结构化错误：步骤 ID、选择器、异常类型、截图路径。
4. 可配置失败策略：`stopOnFail` 或 `continueOnFail`。
5. 默认采用“单窗口 `stopOnFail`”：某个 `iframe` 失败后暂停该窗口，其他窗口继续。

## 5.4 断言引擎（Assertion Engine）

1. 文本断言：元素文本包含/等于。
2. 状态断言：元素存在、可见、禁用状态。
3. URL 断言：跳转目标与参数校验。
4. 结果输出统一格式，便于统计失败分布。

## 5.5 监控与看板（Monitor + Dashboard）

核心指标：

1. 任务级：运行状态、总轮次、成功率、平均轮次耗时。
2. 场景级：步骤通过率、Top 失败步骤、Top 异常类型。
3. 执行器级：活动 `iframe` 数、重建次数、内存趋势（可选）。
4. 系统级：队列积压、心跳丢失、告警次数。

## 6. 长时间运行策略

1. 采用“轮次（Run Loop）”概念，每轮结束都做健康检查。
2. 设置周期性全量重建：例如每 100 轮强制重建全部 `iframe`。
3. 引入心跳机制：每个槽位每 N 秒上报 `alive`。
4. 漂移检测：连续失败超过阈值触发自动降并发和告警。
5. 资源保护：限制最大并发、最大轮次时长、单步超时上限。
6. 默认运行模式为“持续运行”：不设置总轮次上限，直到用户手动停止任务。
7. 任一窗口报错时触发显著告警（看板高亮 + 告警接口预留），并自动暂停该窗口；其他窗口不受影响继续执行。

## 7. 同源与跨域问题（关键风险）

浏览器同源策略决定了：

1. 若 `iframe` 跨域，父页面无法直接访问其 DOM。
2. 录制与断言能力会受限，无法直接执行大部分步骤。

同源与 Agent SDK 的区别：

1. 同源模式：父页面可直接访问 `iframe` DOM，录制/回放实现简单，接入与调试成本低。
2. Agent SDK 模式：页面可跨域，但需在被测页面内注入 SDK，由 SDK 执行命令并通过 `postMessage` 回传结果，接入成本更高但适配范围更广。

建议方案：

1. V1 只支持同源或可注入 Agent SDK 的页面。
2. 对跨域页面，使用页面内 Agent SDK 通过 `postMessage` 与控制台通信。
3. 脚本执行改为“父页面下发命令，子页面 Agent 执行并回传结果”。
4. 本项目 V1 采用“同源 + 反向代理”策略：将不同端口服务统一到同一 Origin（同协议、同主机、同端口），通过路径分发到各业务服务。

本地开发建议拓扑（示例）：

1. 统一入口：`http://localhost:8080`（反向代理网关）。
2. 路径分发：`/app-a -> http://localhost:3001`，`/app-b -> http://localhost:3002`，`/console -> http://localhost:5173`。
3. 录制与回放均从 `http://localhost:8080` 访问，保证父页面与 `iframe` 满足同源条件。

## 8. 数据模型（建议）

1. `projects`
- 项目基本信息。

2. `scenarios`
- 场景定义（页面列表、步骤 DSL、断言规则）。

3. `tasks`
- 执行任务定义（并发、循环策略、失败策略）。

4. `task_runs`
- 每次任务运行实例（开始/结束时间、状态、汇总指标）。

5. `step_runs`
- 每个步骤的执行明细（耗时、结果、错误、截图）。

6. `runner_heartbeats`
- 执行器心跳与健康数据。

## 9. 技术选型建议（可替换）

1. 前端控制台：`React + TypeScript + Vite`。
2. 状态管理：`Zustand` 或 `Redux Toolkit`。
3. 后端服务：`Node.js (NestJS/Fastify)`。
4. 通信：`WebSocket`（实时状态）+ `REST`（配置与历史查询）。
5. 数据库：`PostgreSQL`（结构化数据）+ `Redis`（队列/缓存）。
6. 图表：`ECharts` 或 `uPlot`（长时间趋势更轻量）。

## 10. MVP 分阶段计划

### 阶段 A：最小可跑通（1-2 周）

1. 完成场景 DSL 定义。
2. 完成单 `iframe` 录制 + 回放闭环。
3. 完成任务启动/停止与基础日志。
4. 支持手动设置并发数与持续运行模式。

交付标准：可稳定回放 1 个场景 100 轮。

### 阶段 B：并发与看板（1-2 周）

1. 引入 `iframe` 池并发执行。
2. 完成实时监控面板。
3. 加入重试和失败分类。

交付标准：4-8 并发下可持续运行 8 小时。

### 阶段 C：稳定性增强（1-2 周）

1. 完成健康检查、自动重建、漂移检测。
2. 完成错误归因报告（步骤/页面/异常类型）。
3. 引入基础告警机制。

交付标准：24 小时连续运行，异常可定位。

### 阶段 D：跨域支持（可选）

1. 设计并接入 Agent SDK + `postMessage` 协议。
2. 扩展断言能力和安全校验。

交付标准：跨域页面可执行核心步骤与断言。

## 11. 验收标准（建议）

1. 在目标环境下可并发运行 N 个页面并稳定执行。
2. 录制脚本可重复回放，回放误差可控。
3. 连续运行期间可实时查看任务状态与失败明细。
4. 任意失败均可追溯到具体步骤与页面。
5. 系统异常可自动恢复或触发告警。

## 12. 已确认的关键决策（2026-02-28）

1. 页面接入模式：V1 支持同源页面，或可注入 Agent SDK 的页面。
2. 并发设置：支持手动选择并发数（受系统上限保护）。
3. 运行模式：默认持续运行，直到用户手动停止。
4. 失败策略：默认 `stopOnFail`（单窗口暂停并显著告警，其他窗口继续执行）。
5. 取证能力：首期不做截图/录像，预留二期扩展接口。
6. 部署约束：采用同源方案，使用反向代理统一 Origin；`localhost` 不同端口服务必须经统一入口访问。
